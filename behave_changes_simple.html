PHASE 1
PHASE 1 · MEMORY RELEVANCE

How does the persona pick which memories to recall?

When a persona sees a new ad, we have to choose ~5 memories from potentially hundreds. Picking the wrong ones makes the persona's reaction generic. Opus said our picker was a black box.

BEFORE
We counted how many words from the ad appeared in each memory. More overlap = more relevant. That's it. A Nirma memory and a Surf Excel ad share zero words but should obviously be related.
→
AFTER
A weighted mix of four signals: 40% category match (detergent ↔ detergent), 30% shared themes (nostalgia, value-for-money), 20% era + sensory cues (1990s, jingles), 10% word overlap as a tiebreaker.
Analogy: Before, it was like a librarian who only matches books by counting shared words in the title. Now she also checks the genre, the mood, the era it was written in — and only uses word matches as a final tiebreaker.
PHASE 2
PHASE 2 · MEMORY LIFECYCLE

Memories should fade — and stop growing infinitely.

In the old code, every time a memory got retrieved, its "emotional intensity" went up — forever. Real brains don't work that way: see something often enough and it loses its punch. Old stuff also fades unless something keeps it alive.

Habituation

If a memory gets pulled up more than 5 times in 7 days, its emotional charge stops growing — and starts dampening. Like getting tired of a song.

Frequency cap

The "I've been retrieved a lot" boost caps at 5 retrievals. Without it, one dominant memory would crowd everyone else out forever.

30-day fade

Memories slowly lose intensity over 30 days unless re-triggered. Painful memories fade faster than positive ones — the "fading affect bias" from psychology.

Analogy: Like turning down the volume on a song you've heard 50 times this week, while letting old favourites slowly drift out of rotation unless something brings them back.
PHASE 3
PHASE 3 · CRITIC RULES LAYER

A second pair of eyes — that doesn't need an LLM.

The original critic was one LLM (Sonnet) judging another LLM's output. Opus pointed out: LLMs share blind spots. If Sonnet hallucinates, Sonnet won't catch its own hallucination.

BEFORE — 1 LAYER
Sonnet writes the persona's reaction. Sonnet judges if the reaction is good. Marking your own homework.
→
AFTER — 3 LAYERS
1. Plain Python rules check first (no LLM, no cost): banned phrases, contradictions, vague non-reactions. 2. Sonnet critic. 3. Cross-model GPT-4o-mini sanity check.

What the rules layer catches without any AI

  • Vague reactions ("It's nice", "I like it") — not specific enough.
  • Persona contradictions — a budget-conscious persona suddenly raving about premium.
  • Banned phrases — corporate-speak that no real person says.
  • Score sanity — claims "I'd buy this" but gave it 2/10.
PHASE 3.5
PHASE 3.5 · CROSS-MODEL CHECK

Get a second opinion from a different brain.

Even with the rules layer, both writer and judge were Anthropic models. So we added OpenAI GPT-4o-mini as a lightweight cross-checker. Different training data, different blind spots, different "tells" for what looks real.

Writer

Claude Sonnet

Generates the persona's reaction.

Same-model critic

Claude Sonnet

Catches reasoning errors in detail.

Cross-model critic

Catches stylistic blind spots Sonnet shares with itself.

Why this matters: if both critics flag the same output, that's strong signal. If they disagree, that's signal too — we surface it for human review instead of silently shipping.
PHASE 4
PHASE 4 · PERSONA TENSIONS

Real people contradict themselves. Now our personas can too.

Opus pointed out: a real Indian middle-class buyer might say "I only buy Indian brands" but also own an iPhone. Our personas were too clean — no internal conflicts.

"Tensions" field

Each persona can now declare contradictions like: "Aspires to global brands but feels guilty buying foreign." The persona evaluator actively surfaces these when scoring an ad.

"Non-negotiables"

Hard limits the persona will not cross — e.g. "won't pay more than ₹500 for skincare." The critic flags any reaction that violates these.

Analogy: Before, every persona was a one-page CV. Now they're a CV plus a therapist's note about their contradictions. Reactions feel like a person, not a stereotype.
PHASE 5
PHASE 5 · DOCS HONESTY

Stop overclaiming what the code does.

The original deep-dive HTML said things like "memory gets stronger with use" — true, but it didn't mention the cap, the habituation, the fade. Opus called this out as marketing copy, not engineering docs.

ClaimWhat we saidWhat we say now
Memory strength "gets stronger with every use" "gets stronger up to a cap of 5, then plateaus; habituation kicks in after 5 uses in 7 days"
Manufacturing Nostalgia presented as a measurement framed as research-grounded priors (Rubin & Berntsen 2003) — calibration still open
Critic architecture "LLM critic" "3-layer: deterministic rules → Sonnet → GPT-4o-mini cross-check"
PHASE 6
PHASE 6 · CRITIC BENCHMARK

Don't claim the critic works — measure it.

We built a small "golden set" of 18 hand-labeled examples and ran the deterministic critic against it. No LLM calls — runs in milliseconds, costs $0. Now any future change that breaks the critic will fail the test suite immediately.

10 bad examples

Made-up brand facts, persona contradictions, vague non-reactions, banned corporate phrases. The critic must catch all of these.

8 good examples

Specific, grounded, persona-consistent reactions. The critic must not flag these as bad (no false alarms).

Cost per run $0.00

10 of 10 bad caught. 8 of 8 good cleared. This is now a regression gate — the bar can only go up from here.

PHASE 7
PHASE 7 · EMBEDDINGS WIRED IN

The flag that did nothing — now does something.

We had a function parameter called use_embeddings=True that looked impressive in the API. Opus opened the code and found it was ignored. Memory nodes had embedding: null. Pure vapor.

BEFORE
evaluate_creative(use_embeddings=True) accepted the flag and silently ignored it. The 10% tiebreaker slot in the relevance formula always used keyword overlap.
→
AFTER
The flag actually triggers an OpenAI embedding call (cached per-text so we never pay twice for the same query). When both query and memory have embeddings, the 10% slot uses true cosine similarity. Otherwise falls back to the old keyword path — full backward compatibility.

What changed

  • Relevance scorer accepts an embedding
  • Evaluator computes & passes one
  • Process-level cache dedupes calls

What didn't change

  • Old code paths still work
  • No OpenAI key? Falls back gracefully
  • All 201 prior tests still pass

Honest caveat

Memories saved before this change still have no embedding stored. For those, the old keyword path runs. New memories pick up embeddings automatically — no backfill done.
